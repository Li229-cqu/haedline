# 系统架构说明

## 1. 架构概述

系统采用前后端分离架构。前端负责用户交互和可视化展示，后端以 RESTful API 形式提供新闻处理、历史记录和统计相关服务。后端负责调用 NLP 能力、执行业务规则、保存处理结果并管理上传与导出文件。

```text
用户浏览器
    │
    ▼
前端应用（frontend）
    │ HTTP / JSON / 文件上传
    ▼
后端 API（backend/app/api）
    ├── 新闻处理服务（services）
    ├── NLP / 规则处理能力
    ├── 数据访问层（db、models）
    ▼
数据库 / 本地运行数据（data）
```

## 2. 前端职责

前端负责提供新闻正文输入、单文件上传、处理结果展示、保存操作、历史记录查看及后续统计可视化等界面。前端不直接执行模型推理或数据库操作，而是通过 API 请求后端服务。

## 3. 后端职责

后端基于 API 接收前端请求，并按处理流程组织文本清洗、关键词抽取、实体识别、新闻要素抽取、摘要生成、标题生成和质量校验。后端同时负责参数校验、文件管理、结果持久化与统一响应格式。

后端目录按职责划分：

- `app/api/`：定义新闻处理、历史记录和统计接口。
- `app/services/`：承载文本处理、生成、校验和文件处理等业务服务。
- `app/models/`：定义数据库模型与接口请求/响应数据结构。
- `app/db/`：管理数据库连接与会话。
- `app/utils/`：提供文本及接口响应等通用工具。

## 4. 数据与文件存储

处理记录及其结构化结果存入数据库。用户上传的原始文件存放于 `data/uploads/`，需要导出的结果存放于 `data/exports/`。这两个目录均属于运行数据，不应提交到版本控制系统。

## 5. MVP 技术边界

第一版仅支持单篇新闻处理闭环，模型能力优先使用预训练模型、规则方法和轻量算法集成实现。不包含从零训练模型、用户认证与权限管理、自动发布、爬虫采集及实时流式处理。
