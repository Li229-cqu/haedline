# 第二阶段开发计划：后端基础业务服务

## 1. 阶段目标

第二阶段在第一阶段工程、数据库和模拟接口的基础上，完成单篇新闻从文本或文件输入，到基础预处理、信息统计、记录入库及历史记录管理的后端业务闭环。

本阶段继续使用 FastAPI、MySQL、SQLAlchemy 和现有四张基础表，不进行大规模目录重构，不删除第一阶段已完成的接口与数据结构。

## 2. 开发范围

### 2.1 文本清洗服务

- 在 `app/services/preprocess_service.py` 实现基础文本清洗。
- 处理首尾空白、多余空格、连续空行及常见无效字符。
- 保留新闻正文的主要语义内容。
- 对空文本、清洗后为空及文本超出约定限制等情况提供明确错误信息。

### 2.2 文本统计服务

- 基于原文或清洗后文本计算字数、句子数和段落数。
- 明确中文文本的统计口径，并在接口返回中保持稳定。
- 将统计值保存至 `news_records` 表的对应字段。

### 2.3 文件上传与读取服务

- 在 `app/services/file_service.py` 实现单文件上传、类型校验和内容读取。
- 第一版优先支持纯文本文件；按依赖能力评估 `.docx` 文档读取。
- 将上传文件保存至 `data/uploads/`，使用安全文件名并处理读取失败、空文件和不支持格式。
- 将读取出的正文复用到与手动输入一致的分析流程。

### 2.4 新闻记录入库

- 在新闻处理接口中完成 `NewsRecord` 的创建和保存。
- 保存原始正文、清洗后正文、字数、句子数、段落数和创建时间。
- 返回真实的 `news_id`，供前端跳转详情页和查询历史记录使用。
- 对生成、抽取和质量校验结果表保留关联结构，本阶段允许不创建记录或以空字段保存占位结果。

### 2.5 历史记录查询

- 实现 `GET /api/history` 的真实数据库查询，返回新闻记录列表。
- 第一版提供基础字段，例如记录 ID、原文摘要片段、统计信息和创建时间。
- 支持按创建时间倒序返回；复杂检索、分页和筛选可在后续阶段扩展。

### 2.6 新闻详情查询

- 新增按新闻 ID 查询详情的接口。
- 返回原文、清洗后文本、文本统计信息、创建时间及当前可用的关联结果。
- 对不存在的记录返回明确的 404 错误。

### 2.7 删除新闻记录

- 新增按新闻 ID 删除记录的接口。
- 删除前检查记录是否存在，并向前端返回明确状态。
- 同步处理关联结果记录；具体外键级联策略可在实现时采用清晰、可验证的方式确定。

### 2.8 API 文档更新

- 随接口实现更新 `docs/api.md`。
- 补充文件上传、历史列表、历史详情和删除接口的请求参数、响应结构及错误示例。
- 保持 `NewsAnalyzeResponse` 的字段结构稳定，避免影响前端已有页面。

### 2.9 手动测试清单

- 手动提交正常、空白、超短和多段落新闻文本。
- 验证清洗结果与三项文本统计结果。
- 上传支持类型、空文件、损坏文件和不支持类型文件，验证错误提示。
- 验证成功入库后可在历史列表查看，并可读取对应详情。
- 验证删除后列表和详情查询的结果。
- 使用 Swagger 文档、浏览器或 Postman 验证各接口状态码与响应格式。

## 3. 本阶段明确不实现的功能

第二阶段暂不实现摘要生成、标题生成、关键词抽取、实体识别和一致性校验模型。相关字段继续保留在接口和数据库结构中，当前仅返回模拟字段、空列表、空字符串或空值，以保证前后端接口结构稳定。

不包含批量新闻处理、结果导出、统计分析页面、多文档摘要、用户登录与权限管理等扩展能力。

## 4. 推荐实施顺序

1. 实现文本清洗与文本统计服务，并编写可复用的单元测试或手动样例。
2. 改造单篇新闻分析接口，接入清洗、统计和新闻记录入库。
3. 实现历史记录列表、详情查询与删除接口。
4. 实现文件上传、内容读取并复用单篇处理流程。
5. 更新接口文档，按手动测试清单完成联调与回归检查。

## 5. 阶段完成标准

- 单篇新闻文本可经过清洗和统计后成功保存到 MySQL。
- 历史列表、新闻详情和删除接口均可正常使用。
- 支持的单文件可读取并进入相同处理流程。
- 现有前端首页能够继续调用稳定的分析接口结构。
- 文档、接口示例和手动测试记录与实际实现一致。
